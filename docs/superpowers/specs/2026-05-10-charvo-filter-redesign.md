# CharcoalX Frontend Redesign

## Context

`charvo-filter` is a Next.js 16.2.6 + React 19.2.4 + Tailwind 4 storefront for CharcoalX, a harm-reduction cigarette-filter brand. Five pages exist (`/`, `/products`, `/login`, `/register`, `/orders`) with four custom components (`Button`, `Input`, `SiteHeader`, `SiteFooter`). The current dark warm-charcoal palette (amber `#c08552` on near-black) is on-brand but the component layer is ad-hoc and there is no spec for extending it.

A separate React 19 / Next 16 cascading-render error in `SiteHeader` was fixed prior to this spec (switched from `useEffect` + `useState` to `useSyncExternalStore` against `localStorage`). That fix is the precedent for browser-state patterns in this codebase.

## Goal

Elevate the storefront to a coherent, production-quality design system using shadcn/ui primitives as the spine, with two curated 21st.dev blocks for distinctive hero/product-grid moments. Add two missing pages (Science, Product detail). Refine palette and add fluid typography. Keep the somber harm-reduction tone. Document everything in `docs/frontend-update.md` so future contributors can extend without re-reading the spec.

## Non-goals

- Cart + checkout flow.
- About page / brand narrative pages beyond Science.
- Light mode.
- Real product imagery (continue with inline SVG / lucide icons; defer photography).
- CMS integration (mocked product data remains).
- Mobile-app surface.

## Constraints

- **Next.js 16 + React 19**: cascading renders on `setState` inside `useEffect` are an error. Browser/external state must use `useSyncExternalStore` or be derivable from props/render.
- **Tailwind 4**: tokens are declared in `app/globals.css` via `@theme inline`, not in a JS config.
- **No backend changes**: all redesign work is client-side and template-only. Auth, orders, and product data continue to come from the existing `lib/api.ts` surface.
- **Frappe/Espresso** influence is structural only (spacing scale, fluid type, accessible contrast, component composition discipline). Frappe UI itself is Vue and is not imported.
- **Asset strategy**: prefer registry imports (shadcn, 21st.dev, lucide). Generate net-new assets only when the registry has nothing usable.
- **Package manager**: project uses `npm` (`package-lock.json` present). All install commands documented as `npx` / `npm`.

### Boundaries

- **Tokens** live in `app/globals.css` only. No hard-coded hex in component files.
- **Primitives** (`components/ui/*`) are owned by the shadcn CLI — re-runnable, do not edit by hand except for token-aware className adjustments documented in `frontend-update.md`.
- **Composed components** (`SiteHeader`, `SiteFooter`, `FAQAccordion`, `ScienceDiagram`) are project-owned and may import from `components/ui/*` and from `lib/*` only.
- **Pages** import from `components/*` and `lib/*`. Pages do not own design tokens.

### Component-API replacement

The existing `components/Button.tsx` and `components/Input.tsx` are deleted. Callers are migrated to the new `components/ui/button.tsx` and `components/ui/input.tsx`. If any caller relied on a prop the shadcn versions don't expose (e.g. a custom `loading` prop), the prop is reproduced inline at the call site or via a small wrapper added in the same commit — no orphaned wrappers left behind.

## Component install plan

1. `npx shadcn@latest init` — accept dark mode default, base color "zinc", CSS variables yes, `@/*` alias yes.
2. After init, manually edit the generated `components.json` and CSS to align with tokens above (the init will overwrite `globals.css` — port the redesign tokens back in afterward and commit immediately so the diff is captured).
3. `npx shadcn@latest add button input label card sheet tabs accordion badge skeleton separator sonner form`
4. `npm i lucide-react`
5. `npm i react-hook-form zod @hookform/resolvers`  *(required by shadcn `form`)*
6. `<Toaster />` added once in `app/layout.tsx` (above `</body>`).

## 21st.dev block selection

Two blocks, picked at install via `mcp__21st-magic__21st_magic_component_inspiration` and `mcp__21st-magic__21st_magic_component_builder`:

- **Hero** — centered, eyebrow chip + headline + dual CTA, dark variant. Adapted: copy from current hero, accent recolored to `--accent`, decorative SVG retained as the right column or replaced with the block's preferred illustration if it reads on-brand.
- **Product-grid** — 3-up cards, image area + title + price + CTA, dark variant. Adapted: image area becomes a tinted SVG placeholder until real photography exists, prices via `lib/api.ts:formatPrice`, CTA links to `/products/[slug]`.

Block IDs are recorded in `frontend-update.md` after install with the source URL. If neither block fits cleanly, fall back to hand-composed sections with shadcn primitives — do not import a poor-fit block.

## Page treatments

| Route | Sections (top to bottom) |
|---|---|
| `/` | Hero (21st.dev) → Feature trio (3× shadcn `Card`) → "Honest answers" (`FAQAccordion`) |
| `/products` | Heading + filters bar (later) → Product-grid block → `Skeleton` placeholders during load → empty state if zero products |
| `/products/[slug]` | Two-column on ≥md: gallery (inline SVG) + spec column. Spec column has `Tabs`: Overview / Spec / Honest answers. Sticky bottom add-to-cart on mobile. Related products row at the bottom. |
| `/science` | Long-form prose. Sections: "What activated carbon does", "What it does not do" (boxed `Card` with `border-danger`), "How we test", References list. Inline `ScienceDiagram` (SVG, no external dependency). |
| `/orders` | Empty state CTA when none. Otherwise `Card` per order, `Badge` for status, `Separator` between, click-through to per-order detail (deferred to a later spec). |
| `/login`, `/register` | shadcn `Form` (RHF + zod). Inline error text below each field. Submit button shows `Loader2` icon while pending. Toast on success/failure via Sonner. |

## Data flow

No new server endpoints. All pages continue to call `lib/api.ts`. The product detail page reads `params.slug`, calls a new `getProduct(slug)` in `lib/api.ts` (mocked against the existing product list — added in this same change). Orders page reads `getOrders()` (already exists per current `/orders/page.tsx`).

Browser state (auth user) flows through `useSyncExternalStore(subscribeAuth, getStoredUserSnapshot, () => null)` — the pattern set by the SiteHeader fix. Any new component reading auth uses the same hook; `getStoredUser()` (the eager parse) is reserved for non-render callers.

## Error handling

- **Auth forms**: zod validation surfaces errors inline; API errors caught and rendered as Sonner toast + inline message under the submit button.
- **Product detail not found**: render Next 16 `notFound()` from the page component.
- **Orders empty**: empty-state component, not an error.
- **API failures elsewhere**: try/catch at the page level, render an error `Card` with retry button. Do not surface raw error messages.

## Testing

Manual verification checklist (no automated UI tests in scope this round):

- [ ] `npx tsc --noEmit` passes.
- [ ] `npm run lint` passes.
- [ ] `npm run build` passes.
- [ ] Dev server runs; every route renders without console errors or hydration mismatches.
- [ ] Auth flow: register → login → header shows user + `/orders` link → logout → header reverts. No cascading-render warnings.
- [ ] Product detail at a valid slug renders; invalid slug returns Next 16 not-found.
- [ ] Lighthouse on `/` and `/products` ≥ 90 across categories (deferred if blocked by network).
- [ ] Reduced-motion preference respected (no `tw-animate-css` animations on `prefers-reduced-motion`).
- [ ] Tab order is sensible on every form; visible focus rings everywhere using `--ring`.

## Frontend update doc

`docs/frontend-update.md` is written as part of this work and contains:

1. **What changed** — short narrative, links to this spec.
2. **Tokens** — the table of CSS variables, what each is for, contrast notes.
3. **Component inventory** — primitives (with shadcn registry URL), custom components, imported 21st.dev blocks (with source URL + adaptation notes).
4. **Adding a new shadcn component** — exact `npx shadcn@latest add` command and where it lands.
5. **Swapping a 21st.dev block** — file path, what was adapted, how to re-import a different block without breaking the page.
6. **Conventions** — `useSyncExternalStore` for browser state, no `setState` in `useEffect` bodies, no hex outside `globals.css`.
7. **Known limitations** — mocked product data, deferred cart/checkout, deferred About, no real photography, no light mode.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| `shadcn@latest init` overwrites the carefully refined `globals.css`. | Run init first on a clean commit; then re-apply tokens; commit immediately so the diff is reviewable. |
| 21st.dev blocks ship with their own theme variables that fight `globals.css`. | At import time, rewrite the block to use the project's CSS variables; keep the block in a single file so future swaps are isolated. |
| `react-hook-form + zod` adds runtime weight. | Acceptable — auth is the only forms surface, and it's behind a route the user opts into. |
| Existing callers of `Button`/`Input` rely on undocumented props. | Audit call sites in the same PR as the deletion; add inline wrappers only where prop parity matters. |
| Tailwind v4 + shadcn alignment drift over time. | `frontend-update.md` records exact versions installed and the upgrade procedure. |

## Out of scope (explicit)

- Cart, checkout, payments.
- About / Our story page.
- Light mode toggle.
- AI-generated imagery for hero or product cards.
- Per-order detail page (`/orders/[id]`) — orders list links are present but the destination is a follow-up spec.
- Automated visual regression / Playwright tests.
- i18n.
